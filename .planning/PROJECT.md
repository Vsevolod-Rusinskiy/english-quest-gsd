# English Quest

## What This Is

English Quest — интерактивное веб-приложение для практики английского языка ребёнком уровня Intermediate (материал English File Intermediate, юнит 1A). Ребёнок проходит короткий урок (теория + задания), приложение гибко проверяет ответы, ведёт прогресс, повторяет слабые темы и начисляет игровые «рубли»; родитель получает короткий отчёт после урока.

## Core Value

Проверить механику обучения целиком: детерминированная проверка ответов + LLM-агенты там, где нужна интерпретация, персонализация по прогрессу, повторение слабых тем, начисление бонусов — без единого «сломанного» состояния, даже если агент недоступен.

## Requirements

### Validated

- ✓ Ребёнок видит блок теории (правило + пример) и может отметить «понятно» / «не понятно» — Phase 1
- ✓ Ребёнок проходит упражнения `Lesson-1A.json`: `text-input`, `single-choice`, `matching`, `order-builder`, все 4 типа детерминированно проверяются ядром (без агента) — Phase 1. Оговорка: `single-choice`/`order-builder` в реальном `Lesson-1A.json` контента не имеют (только `text-input`×18 + `matching`×1) — механика проверена на hand-authored фикстурах и живьём в браузере, но ждёт реального контента этих двух типов
- ✓ Ядро нормализует и точно сравнивает `text-input` ответы с `acceptedAnswers` (без агента, exact-match путь) — Phase 1
- ✓ Прогресс (`studentProfile`, `lessonHistory`, позиция в уроке) сохраняется в `localStorage` под ключом `english-quest-progress-v1` синхронно после каждого действия и переживает перезагрузку страницы — Phase 1, подтверждено вживую в браузере
- ✓ Ядро ведёт счётчики попыток/правильных/ошибок/серий и применяет пороговые правила статуса темы по машине состояний (Не изучено → В процессе → Повторить → Выучено, 3 правильных подряд = мастерство) — Phase 2
- ✓ Слабые темы (2+ ошибки) добавляются в `reviewQueue`; ребёнок проходит повторение в той же сессии отдельным проходом после основных 19 упражнений, прогресс переживает перезагрузку — Phase 2
- ✓ Ядро начисляет игровые рубли по фиксированным суммам и лимитам (honest_attempt, first_try_correct, correct_after_hint, streak_bonus, weak_topic_closed), ведёт леджер `rewardHistory` с дедупликацией «без фарма» — Phase 2. Оговорка: `fixed_mistake` определён в схеме, но осознанно никогда не начисляется — свёрнут в `correct_after_hint`, чтобы не платить дважды за одно и то же восстановление (задокументировано и покрыто тестом)
- ✓ При неоднозначном `text-input` ответе вызывается агент Answer Checker с типизированной ошибкой (11-значный enum `errorType`), fallback на строгое сравнение (`errorType: unknown`) — Phase 3. Ядро проверяет `isCorrect:true` от агента по порогу `confidence >= 0.8` перед тем как ему довериться (не принимает вердикт агента вслепую — добавлено после код-ривью, CR-02)
- ✓ Ребёнок может запросить упрощение объяснения теории через агента Theory Tutor до `maxSimplifyRounds` (3) раз; раунд 1 — заранее написанный уровень без агента, раунды 2-3 — агент, после лимита мягкий переход к практике; fallback — повтор последнего заранее написанного уровня, агент никогда не выдумывает текст — Phase 3
- ✓ Общий Agent Gateway (`callAgent()`): validate → retry-once → fallback, единая точка Zod-валидации для обоих агентов Phase 3 (Answer Checker, Theory Tutor); каждое событие фиксирует источник (`core`/`agent`) и факт сбоя — Phase 3. Переиспользуется без изменений остальными 3 агентами в Phase 4

### Active

- [ ] Агент Reward Advisor предлагает причины и текст похвалы; ядро проверяет и начисляет суммы само (агент не решает числа) — Phase 4
- [ ] Агент Progress Advisor даёт следующий фокус, сложность (easy/normal/challenge) и совет по завершению сессии на основе `topicStats`/`wordStats`/`exerciseTypeStats` — Phase 4
- [ ] Агент Parent Report Generator формирует короткий отчёт родителю (пройдено/верно/слабые темы/рубли/рекомендация) с шаблонным fallback — Phase 4
- [ ] Визуальный стиль: детский, блочный, яркий, крупные скруглённые кнопки (Roblox-вдохновлённый, без брендинга Roblox) — Phase 5

### Out of Scope

- Голос, аудио, speaking, микрофон — не входят в MVP (post-MVP)
- Listening, reading как отдельный режим — не входят в MVP
- Автоматический разбор PDF учебника, полный импорт Test Book — данные урока уже готовы вручную как `Lesson-1A.json`
- Шаг «утверждения взрослым» контента урока — `requiresAdultApproval: false` в MVP
- Магазин/аватары за рубли — post-MVP
- Несколько учеников / профилей — `studentId` фиксирован как `primary` в MVP
- Генерация новых заданий агентом (Exercise Generator) — post-MVP
- Сложная мультиагентная оркестрация — оркестратор в MVP это само детерминированное ядро, вызывающее 5 простых агентов-«функций» в фиксированных точках
- Серверная часть / бэкенд — MVP работает полностью локально в браузере

## Context

Проект делается как дипломная работа; вторая явная цель (наравне с самим продуктом) — практика построения архитектуры с AI-агентами, поэтому границы «ядро/агент» и fallback-механизмы являются частью того, что оценивается, а не только конечный UX.

Входные данные урока полностью готовы заранее (`Lesson-1A.json`, схема `lesson-json-v1`) — 3 раздела, 19 упражнений, 26 действий ребёнка (Grammar: present simple/continuous + action/non-action verbs — 9 упражнений; Food vocabulary — 9 упражнений; Restaurant matching — 1 упражнение на 8 пар). Данные не парсятся из PDF в MVP — это осознанно исключено.

Интерфейс смешанный: инструкции на русском, сами упражнения — на английском. Два типа пользователей: ребёнок (основной, проходит урок) и родитель (видит отчёт после урока).

## Constraints

- **Архитектура**: гибрид детерминированного ядра и LLM-агентов — числа и запись состояния только у ядра, агент предлагает суждение/текст, никогда не пишет числа напрямую
- **Хранение**: только `localStorage`, один ключ `english-quest-progress-v1`, без бэкенда
- **Агенты**: ровно 5 независимых агентов-«функций» (Answer Checker, Progress Advisor, Reward Advisor, Parent Report Generator, Theory Tutor) — один запрос → строгий JSON, без сложной оркестрации между собой
- **Отказоустойчивость**: у каждого агента обязателен детерминированный fallback — урок не должен ломаться при недоступности LLM
- **Стек**: TypeScript + Vite + Vitest + Zod, без UI-фреймворка (vanilla DOM, `createElement`/`textContent`, `innerHTML` запрещён репозиторий-wide) — решено в Phase 1 на основе research/STACK.md, компилируется в статический HTML/CSS/JS бандл без сервера
- **Данные первого урока**: фиксированы в `Lesson-1A.json`, схема `lesson-json-v1` — реализация должна соответствовать этой схеме, а не изобретать свою

## Key Decisions

| Decision | Rationale | Outcome |
|----------|-----------|---------|
| Гибрид: детерминированное ядро + LLM-агенты, принцип «агент предлагает — ядро записывает» | Числа (баланс рублей, счётчики, статусы) никогда не должны приходить «из головы» LLM — предсказуемость и тестируемость поверх обучающей механики | ✓ Good — Phase 3, но потребовало явного порога: см. отдельную строку про confidence gate ниже (изначальная реализация без порога нарушала этот же принцип, поймано код-ривью) |
| Входные данные урока готовы заранее (`Lesson-1A.json`), без авто-парсинга PDF в MVP | Сфокусировать MVP на проверке механики обучения, а не на конвейере подготовки контента | ✓ Good — но реальных данных для `single-choice`/`order-builder` в файле нет (см. Validated) |
| Прогресс хранится только в `localStorage`, без бэкенда | MVP локальный по духу — минимизировать инфраструктуру, ускорить проверку гипотезы | ✓ Good — Phase 1, подтверждено вживую (reload, tamper-reset) |
| Ровно 5 простых агентов-функций без сложной оркестрации | Снизить риск непредсказуемого поведения мультиагентной системы в дипломном MVP; каждый агент — независимый строгий JSON-контракт | ✓ Good — Phase 3, 2/5 живых (Answer Checker, Theory Tutor) на одном общем `callAgent()`, без изменений в самом gateway между агентами |
| У каждого агента — детерминированный fallback | Урок не должен ломаться при недоступности/сбое LLM | ✓ Good — Phase 3, оба live-агента: fallback никогда не выдумывает текст (Theory Tutor повторяет последний уровень), не крашит урок |
| Answer Checker: `isCorrect:true` от агента доверяется только при `confidence >= 0.8`, иначе трактуется как неверный ответ (errorType/hintRu всё равно передаются ребёнку) | Код-ривью Phase 3 (CR-02) поймал, что агент мог напрямую подтверждать неверный ответ без порога — прямое нарушение «агент предлагает, ядро решает»; порог 0.8 — продуктовое решение пользователя (строгий, а не 0.6) | ✓ Good — Phase 3, исправлено до сдачи фазы, покрыто регрессионными тестами |
| TypeScript + Vite + Vitest + Zod вместо голого JS из SPEC.md §4 | SPEC.md литерально называет plain JS, но детерминированное ядро — самая оцениваемая часть MVP; типизация и юнит-тесты снижают риск именно там. Компилируется в тот же статический бандл, ограничение «browser-only, no backend» не нарушено | ✓ Good — Phase 1, 68 тестов, чистый typecheck |
| `single-choice`/`order-builder` JSON-схема спроектирована, а не взята из реальных данных (`Lesson-1A.json` содержит только `text-input`+`matching`) | В реальном уроке этих типов физически нет; схему пришлось задать самостоятельно, проверить на hand-authored фикстурах | ⚠️ Revisit — работает (подтверждено в браузере), но ждёт реального контента для полной уверенности |
| Order-builder: tap-to-append/tap-to-remove, без drag-and-drop | Надёжнее для ребёнка на разных устройствах (touch/mouse), драг-энд-дроп добавляет хрупкости без выигрыша | ✓ Good — Phase 1, подтверждено вживую |
| Review-queue проход — отдельный шаг после основных 19 упражнений, не вперемешку | Сохраняет линейную модель навигации `currentExerciseIndex` из Phase 1 без переупорядочивания на лету; соответствует формулировке SPEC «ещё не пройденное в сессии» | ✓ Good — Phase 2, курсор ревью не мутирует основной массив упражнений, переживает перезагрузку |
| Дедупликация наград через сканирование `rewardHistory` по паре (exerciseId, reason), а не отдельный флаг | Единый источник правды вместо параллельного состояния для защиты от фарма | ✓ Good — Phase 2, покрыто тестами |
| Не изобретать механику «мини-тренировки» при 3+ ошибках — эффект уже даёт растущий `reviewQueue` | SPEC не определяет конкретную механику; строить неспецифицированный UI-функционал — это scope creep, у Phase 2 нет бюджета на UI | ✓ Good — Phase 2, простой закрытый набор из 4 переходов FSM |
| Code review Phase 2 поймал 2 критичных бага до сдачи: (1) потеря награды при одновременном мастерстве двух тем из-за скалярной (не массив) переменной; (2) `reviewQueue` никогда не переоткрывался для когда-либо правильно отвеченного упражнения из-за проверки по счётчику за всю сессию вместо «последняя попытка» | Именно такие баги детерминированное ядро обязано не допускать — оба исправлены и закрыты регрессионными тестами до мержа | ✓ Good — процесс сработал как задумано |
