<objective>
Lay the typed, tested foundation for Phase 2's progress engine: extend `ProgressStateSchema` with the new `topicStats`, `currentCorrectStreak`, typed `rewardHistory`, and typed `reviewQueue` fields (all required, so a legacy Phase-1 blob correctly resets), and build the two pure functions the per-answer update will compose — the topic-status FSM (`nextTopicStatus`, D-06) and the review-queue population scan (`enqueueReviewItems`, D-02). Also create the Wave-0 test scaffolds these functions and the reward engine (Plan 02) depend on.

This is the thinnest foundation slice of the MVP: after this plan, a developer can construct topic stats and drive the FSM/queue logic directly in tests and see the correct D-06 transitions and D-02 queue population — the raw mastery machinery is real and inspectable, even though it isn't yet wired into `handleAnswer` (that is Plan 02).

Purpose: PROGRESS-01 (per-exercise counters already exist from Phase 1; this adds the parallel per-topic counters), PROGRESS-02 (topic-status state machine), PROGRESS-03 (2+ errors → needs_review + reviewQueue population). Getting the D-01..D-06 logic exactly right in isolated, table-driven unit tests is the highest-value, lowest-risk work in the whole phase.
Output: extended `progressSchema.ts` + `initialState.ts`; new `topicStatusMachine.ts`, `reviewQueue.ts`; a hand-authored multi-topic fixture; three test files.
</objective>
