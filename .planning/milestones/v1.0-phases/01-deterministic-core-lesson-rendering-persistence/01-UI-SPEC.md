# Phase 1 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## ⚠️ Scope Boundary — Read Before Using This Contract

**Phase 1 is explicitly a FUNCTIONAL-rendering phase, not a visual-polish phase.**

Per `01-CONTEXT.md` (Specific Ideas section): *"Phase 1 should render functionally correct, unstyled-or-minimally-styled screens that Phase 5 will skin."* The full kid-friendly, Roblox-inspired visual identity (blocky shapes, bright saturated palette, large rounded chunky buttons, mascot/reward animations) is **Phase 5's job** (requirements `UI-01`, `UI-02` are mapped to Phase 5 only in `REQUIREMENTS.md`'s traceability table — Phase 1 owns none of them).

This contract therefore defines:
- **Structural layout** — what elements exist on screen and in what order/region (so the executor builds the right DOM structure once, which Phase 5 re-skins rather than rebuilds).
- **Interaction contracts** — tap-to-append/tap-to-remove for order-builder (D-05), theory понятно/не понятно buttons, per-exercise-type submission flow, progress indicator behavior. These are mechanics, not visuals, and are locked now so Phase 5 doesn't have to re-derive them.
- **Baseline visual treatment only** — enough spacing/typography/contrast to be legible and testable by a human during UAT. Neutral, system-default-adjacent styling. No color palette beyond functional feedback (correct/incorrect), no custom iconography, no brand identity.

**Downstream consumers (planner, executor, checker) must NOT interpret this document as requiring the Roblox-inspired kid-friendly skin.** If a task in this phase's plan starts describing custom illustrations, mascots, celebratory animations, or a bright saturated brand palette, that task belongs in Phase 5, not Phase 1. The `gsd-ui-checker` should validate this document against baseline usability/legibility, not against `UI-01`/`UI-02`'s full visual identity bar.

**No design system tooling for this phase.** The project is vanilla TypeScript + Vite with **no UI framework** (per `01-RESEARCH.md`, D-01, and `ARCHITECTURE.md` Anti-Pattern 1 — deliberate "no framework" decision). shadcn is a React/Radix-based tool and does not apply here; the shadcn initialization gate is explicitly skipped for this reason, not skipped by omission. Phase 5 may reconsider tooling for the visual skin layer if needed, but that is out of this document's scope.

## Spacing Scale

Declared values (must be multiples of 4), applied via plain CSS (no framework/utility classes):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Gaps between inline elements (e.g. word-bank chip internal padding) |
| sm | 8px | Compact spacing: gap between order-builder/single-choice chips, hint text to input field |
| md | 16px | Default spacing: card internal padding, gap between prompt and input, gap between top bar and lesson title |
| lg | 24px | Section padding: gap between lesson title and task card, gap between task card and "Проверить" button |
| xl | 32px | Layout gaps: gap between top bar and main content area |
| 2xl | 48px | Not used in Phase 1 (reserved for Phase 5 hero/celebration spacing) |
| 3xl | 64px | Not used in Phase 1 |

**Touch target exception:** order-builder word-bank chips and single-choice option buttons are tappable — minimum 44×44px hit area regardless of visible content size (accessibility/mobile-usability baseline, not an 8-point scale value itself, but the *padding that achieves it* — e.g. `8px` visible padding around a small chip — must come from the scale above).

Exceptions: none beyond the 44px touch-target floor noted above.

## Typography

Baseline legibility only — Phase 5 may swap font family/weights, but must preserve size/line-height ratios established here so text reflow is predictable.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body (prompt text, hint text, RU/EN instructions) | 16px | 400 (regular) | 1.5 |
| Label (button text, progress indicator, chip text) | 14px | 600 (semibold) | 1.2 |
| Heading (lesson title, section title) | 20px | 600 (semibold) | 1.2 |
| Display (theory rule headline, "задание N из 19" if visually emphasized) | 28px | 600 (semibold) | 1.2 |

Font stack: `-apple-system, BlinkMacSystemFont, "Segoe UI", Roboto, sans-serif` (system default — legible, zero network dependency, appropriate placeholder ahead of Phase 5's brand font).

**Bilingual text note (RU + EN instructions per SPEC.md §15):** both languages render at the same Body size (16px/400/1.5) — no visual hierarchy differentiates RU from EN in Phase 1; they are sequential lines in the same task-card block. Phase 5 may differentiate RU vs EN styling; not required here.

## Color

Phase 1 uses color **only for functional feedback and basic UI chrome** — no brand palette. This is a deliberate minimal set, not a placeholder for "eventually pick real colors."

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#FFFFFF` (white) | Page background, task card background |
| Secondary (30%) | `#F3F4F6` (light neutral gray) | Top bar background, theory block background, word-bank/option-chip default (unselected) background |
| Accent (10%) | `#2563EB` (neutral blue) | Reserved for: primary "Проверить" (Submit) button, "понятно" button when active/selected, order-builder chip when placed in sequence, single-choice option when selected — i.e. exactly the elements representing the user's current/primary action target. Not used for decoration. |
| Destructive/Error | `#DC2626` (red) | Reserved for: incorrect-answer feedback banner border/icon only. Never used for buttons the child must tap (no destructive *actions* exist in Phase 1 — see Copywriting Contract). |
| Success | `#16A34A` (green) | Reserved for: correct-answer feedback banner border/icon only. (Second functional signal color, not part of the strict 60/30/10 brand split — feedback states are transient, not layout chrome.) |

Accent reserved for: primary submit button, active/selected state on понятно/не понятно toggle, selected single-choice option, placed order-builder chips. Nothing else uses accent color in Phase 1 (no accent borders, no accent text, no decorative accent) — this keeps the functional screens visually quiet, consistent with the "minimally-styled, Phase 5 will skin" scope boundary.

## Copywriting Contract

All copy in Phase 1 is **plain, calm, functional Russian** (matching SPEC.md's bilingual instruction pattern where applicable) — no personality/mascot voice yet; Phase 5 owns tone polish per its own requirement ("wrong answers presented with non-punitive, encouraging tone").

| Element | Copy |
|---------|------|
| Primary CTA (exercise submit) | "Проверить" (per SPEC.md §15 literal spec) |
| Theory — understood toggle | "Понятно" |
| Theory — not-understood toggle | "Не понятно" |
| Progress indicator | "Задание {N} из {total}" (e.g. "Задание 5 из 19") — per EXERCISE-05, denominator reads from lesson data length, not hardcoded (per RESEARCH.md Open Question 2) |
| Correct-answer feedback | "Верно!" — plain confirmation, no reward/ruble UI yet (rewards are Phase 2 scope) |
| Incorrect-answer feedback | "Не совсем. Попробуй ещё раз." followed by `hint.firstError` text from the lesson JSON if a second attempt is needed — calm, non-punitive by default even in Phase 1's minimal styling (this is a copy/tone requirement, independent of visual polish, and costs nothing to get right now) |
| Empty/loading state (lesson fetch in progress) | "Загружаем урок…" |
| Error state (lesson validation failure, D-06) | Heading: "Не удалось загрузить урок." Body: plain-language restatement of the Zod validation failure (via `z.prettifyError`) plus "Обновите страницу или обратитесь к разработчику." — must never show a raw stack trace or raw Zod error object to the child/parent |

If a "restart lesson" affordance is added by the executor for testing convenience, it is a dev-only control, not a spec'd user-facing destructive action, and is out of this contract. |
| Order-builder — word bank label | "Слова:" (small label above the tappable word-bank chips) |
| Order-builder — sequence area label | "Твой ответ:" (small label above the assembled-sequence area) |
| Matching — instruction reinforcement | Rendered directly from lesson JSON's `instructionRu`/`instructionEn` fields — no additional Phase 1 copy invented beyond what `Lesson-1A.json` already supplies |

## Structural Layout Contract

*(Non-template addition — required by this phase's scope note to make the "what elements exist and where" contract explicit and unambiguous for the executor.)*

Screen regions, top to bottom, present on every lesson screen (theory step and all exercise steps alike):

1. **Top bar** (region, secondary background): lesson/unit title (`unitTitle` from lesson JSON) + progress indicator ("Задание N из total"). Ruble balance and reward chrome are explicitly **out of scope for Phase 1** (Phase 2/5 concern) — do not reserve visual space for it yet beyond what naturally fits; do not stub a fake "0 ₽" display.
2. **Lesson title block** (region, dominant background): section title (`sections[].title`) — shown once per section, or per-exercise if simpler for Phase 1's implementation; executor discretion, not a locked decision.
3. **Main content area** — one of:
   - **Theory screen** (shown once, before first exercise): rule text (`theory.rule`), example (`explanationLevels[0].exampleRu`), "Понятно" / "Не понятно" buttons side by side.
   - **Exercise task card** (dominant background, card-like container with `md`/`lg` padding): instruction line(s) (`instructionRu` then `instructionEn`, stacked), the exercise prompt, the type-specific input UI (below), and the "Проверить" button.
4. **Feedback banner** (appears after submit, below or replacing the submit button until dismissed/advanced): correct (green accent) or incorrect (red accent) + hint text on incorrect.

### Type-specific input UI (structural, per exercise type)

| Type | Structure |
|------|-----------|
| `text-input` | Single-line text `<input>` bound to the blank in `prompt`, focus on render |
| `single-choice` | Vertical (or wrapped horizontal) list of tappable option buttons, one per `options[]` entry; exactly one selectable at a time; selection visually distinguished via accent color per Color contract |
| `matching` | Two columns: left = `leftItems[]` (images/placeholders per `imagePrompt`), right = `rightOptions[]` (word labels); tap-to-pair interaction — tap one left item then one right item to form a pair; paired items visually marked (accent) and removed from further tap targets; unpaired items remain tappable |
| `order-builder` | Two zones per D-05/Pattern 3: "Слова:" word bank (tappable chips, secondary background, unselected) above "Твой ответ:" sequence area (accent-colored chips in chosen order); tapping a bank chip moves it to the end of the sequence; tapping a sequence chip removes it and returns it to the bank; **no drag-and-drop** |

### Interaction contract notes

- **Theory понятно/не понятно:** both buttons always visible together (not a single toggle) — tapping one selects it (accent state) and, in Phase 1 (no Theory Tutor agent yet), immediately advances to the first exercise per D-06/scope (Phase 3 adds the "не понятно → simpler explanation" loop; Phase 1 only needs the two buttons to exist and be tappable, satisfying THEORY-02 literally without the Phase 3 branch).
- **Submit ("Проверить") button:** disabled/inert until the exercise has a submittable answer (text entered for `text-input`; one option selected for `single-choice`; all pairs made for `matching`; all bank words placed in sequence for `order-builder` — or, at minimum, at least one word placed, executor discretion on exact enable threshold since SPEC.md doesn't specify partial-submission behavior).
- **Progress indicator:** updates immediately on advancing to the next exercise (after feedback is dismissed / "next" is tapped), not before.
- **No animations, transitions, or celebratory effects required in Phase 1** — instant state swaps are acceptable; Phase 5 adds motion polish.
