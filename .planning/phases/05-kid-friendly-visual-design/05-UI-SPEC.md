## Typography

Sizes are the FLOOR from Phase 1 (D-08) — this phase increases Display/Heading for punch, keeps Body/Label ratios intact so text reflow assumptions from Phase 1-4 remain valid. Weight increases from Phase 1's max (600) to include a heavier 700 for the "chunky game UI" feel, replacing 600 as the "bold" role where Phase 1 used semibold.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px | 400 (regular) | 1.5 |
| Label | 14px | 700 (bold — was 600 in Phase 1) | 1.2 |
| Heading | 22px (was 20px in Phase 1 — modest bump) | 700 (bold) | 1.2 |
| Display | 32px (was 28px in Phase 1, within D-08's "up to 32-36px" allowance) | 700 (bold) | 1.2 |

**Rationale for exactly 2 weights (400/700):** matches the design contract's "2 weights max" rule; regular for body copy (readability at length), bold for everything that needs to feel chunky/game-like (headings, labels, button text, display). Phase 1's intermediate 600 weight is retired in favor of 700 everywhere it previously appeared, since D-04's "blocky/chunky" direction calls for a stronger weight contrast against body text, not a subtle one.

**4 sizes total (14/16/22/32):** satisfies "must declare exactly 3-4" — kept at 4 to preserve Phase 1's Body/Label/Heading/Display role separation rather than collapsing roles.

Font stack: unchanged, `-apple-system, BlinkMacSystemFont, "Segoe UI", Roboto, sans-serif` (D-07).

**Bilingual text note (RU + EN instructions, unchanged from Phase 1):** both languages still render at Body size (16px/400/1.5), sequential lines in the same task-card block — Phase 5 does not introduce a visual hierarchy between RU/EN (not required by D-08, avoids scope creep).

## Color

Extends Phase 1's 60/30/10 token structure per D-02 — same CSS custom-property NAMES (`--color-bg`, `--color-bg-secondary`, `--color-accent`, `--color-error`, `--color-success`), new VALUES. Bright, saturated, primary-color-forward per D-01. Exact hex values below are this agent's discretion call (D-01's explicit "left to planner/ui-researcher" note), chosen from standard high-saturation blue/yellow/green/red kids'-app palettes, verified to avoid Roblox's actual brand red (`#E2231A`)/white/black logo mark.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#FFF8E7` (warm off-white / cream, NOT stark white) | Page background, task card background. Rationale: pure white (`#FFFFFF`, Phase 1's value) reads clinical/neutral; a warm cream keeps 60% of the screen "friendly" without introducing a second saturated hue that would compete with the 10% accent — a common technique in bright kids'-app palettes (soft warm neutral base + saturated accents on top). |
| Secondary (30%) | `#DCEEFF` (soft saturated light blue) | Top bar background, theory block background, word-bank/option-chip default (unselected) background, card borders |
| Accent (10%) | `#2E7DF7` (bright saturated blue — bolder than Phase 1's `#2563EB`, same hue family for continuity) | Reserved for: primary "Проверить" button, "Показать итоги"/session-end continue button, "понятно" button when active/selected, order-builder chip when placed in sequence, single-choice option when selected, progress-indicator fill, reward-toast background. Same reserved-for list as Phase 1, extended only to the two new D-10 elements (reward toast, session-end CTA) since those are new surface, not new accent usage on existing elements. |
| Destructive/Error | `#E8544A` (warm saturated red, softer than Roblox's brand red `#E2231A` — deliberately shifted off that exact hue) | Reserved for: incorrect-answer feedback banner border/icon only, per D-03's "used sparingly, never harsh full-bleed" rule. Never a button background — no destructive actions exist in this lesson flow (unchanged from Phase 1). |
| Success | `#3DBB5E` (bright saturated green) | Reserved for: correct-answer feedback banner border/icon only, correct-answer bounce-in accent, reward-toast checkmark icon color |

**Second color family — Yellow (D-01 palette requirement, not in Phase 1's set):** `--color-highlight: #FFC93C` (bright saturated yellow) reserved EXCLUSIVELY for: the reward "+N ₽" toast icon/border and the ruble-balance chip in the top bar (D-10, UI-02) — i.e., money/reward visual language uses yellow/gold consistently, distinct from the blue accent used for primary actions. Not used anywhere else (not a third "brand" color competing with the 60/30/10 split — it's a semantic reward-color, same category as success/error).

Accent reserved for: primary submit button, session-end continue button, active/selected понятно/не понятно toggle, selected single-choice option, placed order-builder chips, progress-indicator fill bar. Nothing else uses accent blue — decorative use is explicitly out (D-04 rejects gradients/decoration beyond the flat drop-shadow noted in Shape Language below).

**Border color technique (D-04 "chunky game UI"):** every colored surface (buttons, cards, chips) gets a 2-4px border in a DARKER shade of its own fill color (not black, not gray) — e.g. accent button `background: #2E7DF7` / `border: 3px solid #1E5FC7`; success banner `border: 3px solid #2C9647` on a light `#E8F9EC` fill. Exact darker-shade formula: reduce lightness ~15-20% from the base hex. This is a CSS-only technique (no filter/blend-mode needed — precomputed hex pairs per role), applied uniformly across every colored component.

## Shape Language & Components

*(Non-template addition — required because D-04/D-06 define a component-level visual system beyond color/type/spacing that the checker/executor need spelled out explicitly.)*

| Property | Value |
|----------|-------|
| Border radius (cards, chips, buttons) | 14px (within D-04's 12-16px range) — applied uniformly, not per-component variation |
| Border width (colored elements) | 3px (within D-04's 2-4px range), darker-shade-of-own-color per Color contract above |
| Border radius exception | none — no fully-circular/pill elements per D-04 ("not fully pill-shaped/circular") |
| Shadow | Single flat drop-shadow on primary CTA buttons only: `box-shadow: 0 4px 0 {darker-border-shade}` (a solid "3D block" offset shadow, NOT a blurred soft shadow — this is the classic "chunky button" technique: same darker shade as the border, zero blur, creates a pressed-in-plastic look). No shadow on cards, chips, or secondary buttons — reserved for primary CTAs only, consistent with accent-color reservation logic (visual weight matches interaction priority). |
| Button press state (D-06) | `:active` — button shifts down 4px (matching the shadow offset, so the shadow appears to "disappear" as if physically pressed) + border-color shifts one step darker. Pure CSS `:active`, no JS. |
| Primary CTA height | 52px (see Spacing Scale touch-target exception) |
| Secondary/chip touch targets | 44px minimum (Phase 1 floor, unchanged) |
| Decorative shapes | Plain CSS shapes only per D-05: circles (`border-radius: 50%`) and squares/triangles (`border` triangle trick) — used sparingly as small accents (e.g. a decorative circle behind the ruble icon), never as illustration |

## Motion Contract

*(Non-template addition — required by D-09/D-10's explicit motion decisions.)*

| Interaction | Motion | Timing (discretion, per D-09) |
|-------------|--------|-------------------------------|
| Correct-answer feedback | Bounce/scale-in: `transform: scale(0.9) → scale(1.05) → scale(1)` | 350ms, `ease-out` |
| Incorrect-answer feedback | Gentle shake: `translateX(-4px, 4px, -2px, 0)`, small amplitude, calm not punitive per ROADMAP SC4 | 300ms total, `ease-in-out`, single pass (no repeat) |
| Agent "thinking" indicator (shared across all 5 agent calls per D-09) | Pulsing/rotating CSS `@keyframes` — a single reusable `.thinking-indicator` class: a small circle that pulses opacity `0.4 → 1 → 0.4` in a loop | 1000ms per cycle, `ease-in-out`, `infinite` — same class/timing reused verbatim for Answer Checker, Theory Tutor, Progress Advisor, Reward Advisor, Parent Report Generator waits (never 5 bespoke animations) |
| Reward "+N ₽" toast (D-10) | Slide-up + fade-in on trigger, auto-dismiss after a hold period, slide-down + fade-out on exit | Enter: 250ms `ease-out`; hold: 1500ms static; exit: 200ms `ease-in` |
| Screen/exercise transitions | None required — instant state swaps remain acceptable (Phase 1's baseline stands; D-09 only mandates motion for the 3 specific cases above, not general page transitions) |

All motion is CSS transitions/`@keyframes` only — no JS animation library, per D-09.

## Structural Additions This Phase (D-12 + UI-02)

Two pre-existing bugs are in scope (D-12), plus one new top-bar element (UI-02) not present in Phase 1's contract:

1. **Ruble balance in top bar (UI-02, NEW):** Phase 1 explicitly deferred this ("do not stub a fake '0 ₽' display"). This phase ADDS it for real: top bar now shows lesson title + progress indicator (unchanged) + ruble balance chip (new), reading live from `state.currentRewards`. Placement: right-aligned in the top bar, opposite the lesson title, on the same row as (or directly adjacent to) the progress indicator.
2. **Feedback banner clearing on exercise-advance (D-12 / Gap 1 fix):** the banner must be cleared (removed from DOM or hidden) the moment the child advances to the next exercise, not left visible until the next submit. This is a logic fix touching `main.ts`'s `feedback` transient state, exposed here because Phase 5's screen rework touches this exact code path anyway.
3. **Progress indicator overshoot fix (D-12 / Gap 2 fix):** at lesson-complete, the indicator must not render "N+1 из N" — it should clamp at "N из N" or switch to a completion-specific visual state (e.g. a checkmark/celebration treatment) rather than overshoot the denominator.
