<objective>
Deliver the shared Agent Gateway trust boundary and the first agent built on it — Answer Checker — as one working end-to-end vertical slice. After this plan, a child who types a near-miss (non-exact-match) text-input answer gets an LLM-assisted verdict with a typed `errorType` and a Russian hint, and if the agent is unavailable the lesson silently falls back to strict comparison with `errorType: "unknown"` — never crashing or stalling.

Purpose: Establish the single validated agent-call path (validate → retry-once → fallback) that RELY-01/02/03 require and that Theory Tutor (Plan 02) and all three Phase 4 agents will reuse unchanged. This is the phase's spine: the async conversion of `handleAnswer` and `main.ts`, and the `source`/`agentFailed` event logging, are established here.

Output: `@anthropic-ai/sdk` installed; `src/core/agents/` module (client, gateway, Answer Checker schema + wrapper); extended `CheckResult`; extended `exercise_attempt` action with `source`/`agentFailed`; async `handleAnswer` + async `main.ts` text-input submit with a "thinking" cue; Wave 0 tests plus async-conversion fixes to the existing suite.
</objective>

## Existing contracts (do NOT break)

- `checkTextInput(exercise, rawAnswer): CheckResult` — pure, synchronous, exact-match only. Its own body stays synchronous and unchanged; only the `CheckResult` interface is extended and the *caller* (lessonEngine) branches on `isCorrect === false`.
- `CheckResult` today: `{ isCorrect: boolean; source: "core" }`.
- `StateStore.dispatch(action)` — synchronous, calls `save()` then notifies listeners. One dispatch = one save = one render (store.ts header). Agent outcomes MUST fold into the existing `exercise_attempt` action, never a new dispatch call site.
- `exercise_attempt` action already carries `topicUpdates`, `reviewQueueAdditions`, `rewardEvents`, `nextCorrectStreak`, `reviewDequeueId?` — add fields additively, following the Phase 2 comment convention.
- `LessonEngine.handleAnswer(exerciseId, answer): CheckResult` — synchronous today; computes `evaluateAttempt` delta then dispatches once. Becomes `async` / `Promise<CheckResult>`; the `await callAnswerChecker(...)` must complete BEFORE `evaluateAttempt` runs so evaluateAttempt still receives a complete final CheckResult.
- `main.ts` `onSubmit` closure — currently synchronous; uses unsubscribe-before / resubscribe-after around `engine.handleAnswer()`. Keep that structure; make the closure `async` and `await` the engine call inside the unsubscribe window.

## New contracts this plan creates (Plan 02 + Phase 4 consume these)

- `callAgent<T>(opts): Promise<AgentCallResult<T> | AgentFallbackResult<T>>` — the shared gateway. `AgentCallResult<T> = { data: T; source: "agent"; failed: false }`, `AgentFallbackResult<T> = { data: T; source: "core"; failed: true }`.
- `anthropicClient` — module-level Anthropic instance. Gateway MUST accept a client via dependency injection (default = `anthropicClient`) so tests inject a stub.
- `AnswerCheckerResponseSchema`, `AnswerCheckerErrorType` (11-value enum), `callAnswerChecker(input): Promise<...>`.
- Extended `CheckResult`: `{ isCorrect: boolean; source: "core" | "agent"; errorType?: AnswerCheckerErrorType; confidence?: number; hintRu?: string }`.
</interface_context>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| child free-text answer → Answer Checker prompt | Untrusted user input is embedded as DATA in `userContent`, never as instruction; system prompt is fixed |
| LLM router (api.llmrouter.ru) → app | Third-party proxy response crosses into the app; treated as untrusted until Zod-validated (extra network hop vs. calling Anthropic directly) |
| agent JSON response → core state write | No agent value reaches state until the shared Zod validation accepts it (RELY-01) |

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-01 | Information Disclosure | API key in browser bundle (`import.meta.env.VITE_LLM_API_KEY`) | high | accept | Explicitly accepted, scoped tradeoff (D-03) — local dev / in-person demo only; already documented in STATE.md Blockers. Do NOT add mitigation code this phase; a proxy swap is the deployment-process fix. No code change required. |
| T-03-02 | Tampering | Prompt injection via child's free-text answer reaching Answer Checker | low | mitigate | Child answer passed as `userContent` DATA (system prompt fixed); the actual boundary is Zod schema + enum-membership validation — even a "successfully injected" response cannot write anything the schema disallows (T-03-03). |
| T-03-03 | Tampering / Elevation of Privilege | Trusting agent output as authoritative without independent validation | high | mitigate | `callAgent` Zod-`safeParse`s every response before returning `source:"agent"` (RELY-01), even with `strict:true` set (defense in depth against router-layer drift). Wrong-enum/malformed → fallback path. This is the single shared validation function (D-05). |
