<objective>
Make an answer *do something*: build the fixed-rule reward engine (`computeRewardEvents`, D-03/D-04/D-05) and the single per-answer aggregator (`evaluateAttempt`) that loops topics (D-01), runs the FSM, populates the review queue, and computes reward events — then wire it into `LessonEngine.handleAnswer()` and `StateStore` so all four state slices (topicStats, reviewQueue, rewardHistory/currentRewards, currentCorrectStreak) fold into ONE enriched `exercise_attempt` dispatch, preserving Phase 1's "exactly one save/render per user action" invariant.

MVP slice check: after this plan, a child playing the real lesson accrues rubles and moves topics through the status machine, and weak topics get queued — observable, persisted state the child could not produce after Plan 01 (which only built the isolated functions). This is the vertical slice that connects the machinery to the live answer flow.

Purpose: REWARD-01 (fixed amounts + per-exercise dedup limits), REWARD-02 (rewardHistory ledger), and it completes PROGRESS-01/02/03 by driving the Plan 01 functions from the real handleAnswer path.
Output: new `rewardEngine.ts`, `evaluateAttempt.ts`; extended `store.ts` (enriched exercise_attempt action + reduce), `lessonEngine.ts` (handleAnswer integration); reward-engine tests + lessonEngine single-dispatch guard test.
</objective>
