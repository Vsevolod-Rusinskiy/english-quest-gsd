<objective>
Route all browser LLM calls through a thin Cloudflare Worker key-proxy instead of calling api.llmrouter.ru directly from the browser. This fixes the CORS-preflight-401 blocker (STATE.md Blockers, diagnosed 2026-07-04: the router rejects the browser's OPTIONS preflight with 401, which curl never sends) and removes the client-bundled-API-key exposure (03-CONTEXT.md D-03) by holding the real router key as a Cloudflare secret, server-side only.

Purpose: Make live agent responses actually observable from a real browser (server-to-server forwarding sidesteps CORS entirely) and make `dist/` safe to deploy publicly (no extractable key in the shipped JS).

Output: A `worker/` directory containing a deployable Anthropic-compatible `/v1/messages` proxy Worker plus `wrangler.toml`, a locally-runnable `wrangler dev` proxy for testing, and an `anthropicClient.ts` whose `baseURL` points at the Worker with no real key in the browser bundle. Actual `wrangler deploy` is a documented manual follow-up (needs the user's Cloudflare credentials).
</objective>

  <action>
Create a self-contained `worker/` subdirectory (kept separate from the Vite app so its Cloudflare/Workers-runtime tsconfig and deps never leak into the browser build).

worker/src/index.ts — a Cloudflare Worker (module-syntax `export default { async fetch(request, env) {...} }`) that acts as an Anthropic-compatible reverse proxy:
- Read the upstream base from `env.LLM_UPSTREAM_URL` (default to the api.llmrouter.ru origin if unset) and the real key from `env.LLM_API_KEY` (a Worker secret / .dev.vars value — NEVER hardcoded, NEVER in wrangler.toml).
- Handle the CORS preflight itself: for `OPTIONS`, return 204 with permissive CORS headers (Access-Control-Allow-Origin, Access-Control-Allow-Methods including POST/OPTIONS, Access-Control-Allow-Headers covering the SDK's headers: content-type, x-api-key, authorization, anthropic-version, anthropic-beta, and any x-* the SDK sends). This is the exact fix for the STATE.md blocker — the router refused the preflight; the Worker answers it without auth, as a CORS-compliant server must.
- For `POST` to a path ending in `/v1/messages`: forward the request body verbatim to `${upstream}/v1/messages` server-side, injecting the real key header(s) the router expects (send BOTH `x-api-key: env.LLM_API_KEY` and `Authorization: Bearer env.LLM_API_KEY` for compatibility, plus pass through `anthropic-version` and `content-type` from the incoming request; do NOT forward any client-supplied auth header — the Worker is the sole key holder). Stream/return the upstream response body and status back to the browser, adding the same permissive CORS response headers so the browser accepts it.
- For any other method/path: return 405 or 404. Keep it thin — NO auth system, NO rate limiting, NO logging infra, NO app logic (per quick-task constraints; the only "backend" allowed is this passthrough).

worker/wrangler.toml — name (e.g. "english-quest-llm-proxy"), main = "src/index.ts", compatibility_date (a recent date). Declare `[vars]` for LLM_UPSTREAM_URL (the non-secret upstream origin, e.g. "https://api.llmrouter.ru"). Do NOT put LLM_API_KEY here — it is a secret set via `wrangler secret put` / .dev.vars.

worker/.dev.vars.example — a committed template documenting `LLM_API_KEY=your-real-router-key-here` for `wrangler dev`. The real `worker/.dev.vars` must be gitignored (Task 3 handles the .gitignore entry).

worker/package.json — minimal, private, with a `dev` script `wrangler dev` and `deploy` script `wrangler deploy`, and `wrangler` as a devDependency (a current 3.x/4.x). worker/tsconfig.json — target the Workers runtime (lib ES2022, module ESNext, types for @cloudflare/workers-types or a hand-declared minimal Env interface if adding the types package is undesirable; a minimal inline `interface Env { LLM_API_KEY: string; LLM_UPSTREAM_URL?: string }` in index.ts is acceptable to avoid an extra dependency).

Do not run `wrangler deploy` — it needs the user's Cloudflare auth (called out as manual follow-up in user_setup).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → Worker | Untrusted browser request (Anthropic-shaped POST) crosses to the Worker; carries no real key. |
| Worker → api.llmrouter.ru | Server-to-server; the Worker injects the real key here (the only place it exists at runtime). |
| key at rest | Cloudflare secret / worker/.dev.vars (gitignored) vs. the previous browser-bundled key. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-rl5-01 | Information Disclosure | Real router API key | high | mitigate | Key moves from the browser bundle to a Worker secret (`wrangler secret put` / gitignored `.dev.vars`); browser sends only a placeholder. Task 2 + Task 3. This closes 03-CONTEXT.md D-03's exposure. |
| T-rl5-02 | Information Disclosure | worker/.dev.vars committed to git | high | mitigate | .gitignore entry for `worker/.dev.vars` + `.dev.vars`; only `.dev.vars.example` (no real key) is committed. Task 1 + Task 3. |
| T-rl5-03 | Spoofing / Tampering | Worker treated as an open relay (anyone can POST) | medium | accept | Diploma-MVP scope explicitly forbids adding an auth system / rate limiting to this thin proxy. Cloudflare's free-tier request cap bounds abuse; the upstream router still enforces its own quota. Documented as accepted for the demo. |
| T-rl5-04 | Elevation of Privilege | Client-supplied auth header forwarded upstream | low | mitigate | Worker forwards ONLY its own injected key header and never passes through any client Authorization/x-api-key header. Task 1. |
