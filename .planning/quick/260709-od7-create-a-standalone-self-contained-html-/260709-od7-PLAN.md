<verified_facts>
Single source of truth for every specific claim in the page. Verified against source during planning.

**Agent Gateway (`src/core/agents/callAgent.ts`)**
- One shared generic function `callAgent<T>()` — reused UNCHANGED by all 5 agents (not 5 separate integrations).
- Model: `claude-haiku-4-5`.
- Per-attempt timeout: `TIMEOUT_MS = 12000` (12s). SDK-level `maxRetries: 0` (gateway does its own retry). `max_tokens: 512`.
- Flow: build request (strict tool-use, `tool_choice` forced to the agent's single tool) → call LLM → filter for exactly 1 `tool_use` block → Zod `safeParse` the tool input → on ANY failure (transport OR Zod), retry EXACTLY ONCE (no backoff) → on second failure, log and return the agent's deterministic `fallback` with `source:"core"`, `failed:true`. Success returns `source:"agent"`, `failed:false`.
- TIMEOUT_MS was raised 8000→12000 after live smoke-testing showed real Worker latency ~6.4–7s too close to the old 8s ceiling.

**Cloudflare Worker proxy (`worker/src/index.ts`)**
- Deployed at `https://english-quest-llm-proxy.leto99999.workers.dev`; forwards to upstream `https://api.llmrouter.ru` (`/v1/messages`).
- Holds the real API key server-side only (`env.LLM_API_KEY`, a Worker secret) — never in the browser bundle.
- Answers the CORS `OPTIONS` preflight ITSELF with `204` and no auth check (the router rejected preflights with 401 — a real live-found bug; curl never sends preflights, real browsers do).
- Thin reverse proxy only: no auth system, no rate limiting, no app logic.
- Two reasons it exists: (a) security — key must never ship in browser JS; (b) it structurally fixes the CORS problem.
- Follow-up live fix: the `anthropic-dangerous-direct-browser-access` header (auto-sent by the SDK under `dangerouslyAllowBrowser:true`) had to be added to the Worker's `Access-Control-Allow-Headers` — it was silently blocking the real POST client-side even though OPTIONS returned 204 and curl worked.

**Answer Checker (`src/core/agents/answerChecker.ts`)**
- Triggered ONLY when a text-input exercise's deterministic exact-match check fails. NEVER called for single-choice/matching/order-builder (those are always deterministic-only, zero agent call).
- Proposes `isCorrect` / `errorType` / `confidence` / `hintRu`.
- Trust gate: `CORRECT_CONFIDENCE_THRESHOLD = 0.8` — the core accepts `isCorrect:true` from the agent ONLY when `confidence >= 0.8`; below that, downgraded to `isCorrect:false` (hint still passed through).
- Fallback: `isCorrect:false`, generic `errorType:"unknown"`, no confidence/hint (the agent never spoke).

**Theory Tutor (`src/core/agents/theoryTutor.ts` + `lessonEngine.handleTheoryStep`)**
- Triggered by the child tapping "Не понятно" on the theory screen.
- `maxSimplifyRounds = 3` (from the lesson data). Round 1 (`simplifyRoundCount === 0`): CORE-ONLY, NO agent call — serves the pre-written `explanationLevels[1]` ("simple") text. Rounds 2–3: call the agent to reformulate even simpler.
- Soft transition: when the incremented round count reaches `maxSimplifyRounds` (3), `theoryUnderstood` becomes true regardless of the last "Не понятно" tap — auto-advances to the first exercise (prevents an infinite loop). This transition currently has NO explanatory UI cue — a known, still-open UX rough edge.
- Fallback: re-serves the SAME pre-written simple text verbatim — NEVER fabricates new text.

**Reward Advisor (`src/core/agents/rewardAdvisor.ts` + cross-check in `lessonEngine.handleAnswer`)**
- Triggered once per answer, ONLY when that answer produced ≥1 reward event (the core's reward rules already decided rubles were earned). The agent NEVER decides the amount — it has no `amount` field — it only proposes `celebrationRu` praise + `suggestedReasons`.
- Anti-hallucination cross-check gate (in lessonEngine): the agent's `suggestedReasons` are intersected (Set intersection) against the ACTUALLY-granted `delta.rewardEvents`. If the agent suggests a reason that didn't actually happen, `praiseRu` becomes `undefined` — identical to an agent failure. `praiseRu` is transient (returned from handleAnswer, never persisted).
- Fallback: no praise text.

**Progress Advisor (`src/core/agents/progressAdvisor.ts`) — called once at session end**
- Proposes `recommendedFocus`, `suggestedDifficulty`, `reviewSuggestions`, `motivationalMessageRu`, `sessionAdvice`.
- TWO core-side guardrails on its output (both real live-found bugs — "trust but verify"):
  (a) `applyDifficultyGuardrails()` (`src/core/personalization/difficultyGuardrails.ts`): the ONLY writer of `difficultyMode`. Caps any jump to exactly ONE step per call (no direct easy↔challenge). Upward move requires `correctStreak >= 3`; downward requires `recentErrors >= 2`; otherwise unchanged.
  (b) `applyRecommendedFocusGuardrail()` (`src/core/personalization/recommendedFocusGuardrail.ts`): validates `recommendedFocus` against the known topic-id key set in `topicLabels.ts`; falls back to the core's own computed weakest-topic if the agent returns anything else. This closed a real live bug where the agent returned a hallucinated mixed-language string instead of a clean topic id.
- Fallback: uses the core's own weakest-topic (highest error/lowest correct ratio in `topicStats`), keeps difficulty unchanged, fixed encouragement message.

**Parent Report Generator (`src/core/agents/parentReportGenerator.ts`)**
- Called immediately AFTER Progress Advisor resolves — SEQUENTIAL, explicitly NOT `Promise.all` — so it can safely use Progress Advisor's already-guardrailed FINAL `recommendedFocus`.
- Proposes `parentReportRu` / `headlineRu`.
- Fallback: `buildTemplateReport()` — a deterministic Russian template from the session snapshot (exercises completed, correct count, struggling/review topics via `topicLabel()`, rubles earned, recommendation). This template was itself the site of a real live bug (it once interpolated raw topic-id strings before `topicLabels.ts` existed) — now fixed.

**Answer-checking data flow (`lessonEngine.handleAnswer`)**
- Deterministic check FIRST, by type: `checkTextInput` / `checkSingleChoice` / `checkMatching` / `checkOrderBuilder`.
- Only text-input escalates to Answer Checker on an exact-match miss; every other type is fully deterministic (no agent).
- `evaluateAttempt()` computes ALL derived state in one pure pass (topic stats + FSM, review queue, reward events, word/exercise-type stats, streaks).
- Reward Advisor conditionally called (if ≥1 reward event) → single `exercise_attempt` dispatch → persisted to `localStorage["english-quest-progress-v1"]` → UI re-renders (feedback banner, top-bar chips, progress bar).

**Topic mastery / review-queue FSM (`topicStatusMachine.ts` + `reviewQueue.ts`)**
- States: `not_started → in_progress → needs_review` (on 2 total errors accumulated) / `mastered` (on 3 correct-in-a-row from any state). Pure core logic, ZERO agent involvement.
- A `needs_review` topic's not-recently-correct exercises are appended to the review queue and re-served later in the SAME lesson session (a review pass after the main pass). A review-pass answer dequeues the item regardless of correctness.
- Note (strict rule, worth a one-line caveat): once a topic has ever accumulated ≥2 lifetime errors, a single later wrong answer snaps it back to `needs_review` even after mastery.

**Reward mechanics (`rewardEngine.ts` + `coin.ts`)**
- Fixed reward-amount table in core (never chosen by an agent): `honest_attempt`=1, `correct_after_hint`=3, `fixed_mistake`=4, `first_try_correct`=5, `streak_bonus`=10, `weak_topic_closed`=15 (rubles, ₽).
- `streak_bonus` (+10) fires when the session-global correct streak reaches `>= 5` (then resets).
- A synthesized "coin clink" sound (`src/ui/sound/coin.ts`, Web Audio API, NO external asset — two bell-like notes B5 ~988 Hz then E6 ~1319 Hz) plays when the ruble balance increases; degrades silently (never throws) if AudioContext is unavailable/blocked.
- A streak chip "🔥 N" shows only when `currentCorrectStreak >= 2`.

**Session-end flow**
- lesson complete → "Показать итоги" tap → Progress Advisor call → guardrails applied (difficulty + recommendedFocus) → Parent Report Generator call (sequential) → ONE `session_end` dispatch → combined session-end screen for both child (motivation + next focus + rubles) and parent (report).

**Thesis framing (CLAUDE.md)**
- Central claim to state verbatim in spirit: "agent proposes, core validates before use" / "agent response is not trusted until the core validates." Every one of the 5 agents has a working deterministic fallback — timeouts, CORS failures, and hallucinated outputs were all observed live in this project's own testing and handled gracefully without crashing.

**Real live-found bugs (STATE.md) — good "guardrail architecture catching a real problem" case-study material**
- CORS preflight 401 → fixed by the Worker answering OPTIONS itself (+ the `anthropic-dangerous-direct-browser-access` header follow-up).
- Raw snake_case topic-id leaking into user/parent-facing Russian text → fixed by `topicLabels.ts`.
- Missing core-side guardrail on `recommendedFocus` (agent returned a hallucinated mixed-language string) → fixed by `applyRecommendedFocusGuardrail()`.
- 8s timeout too tight vs real ~6.4s+ latency → raised to 12s (`TIMEOUT_MS`).
