1 Обзор архитектуры (core/agent split)

▾

Центральный тезис проекта (сформулирован в CLAUDE.md): агент предлагает — ядро проверяет перед использованием. Детерминированное ядро (core) — единственный писатель состояния и единственный владелец всех чисел в приложении. Ровно 5 независимых LLM-агентов-функций — Answer Checker, Theory Tutor, Reward Advisor, Progress Advisor, Parent Report Generator — только предлагают текст или суждение (подсказку, похвалу, рекомендацию), но никогда не пишут состояние напрямую.

Детерминированное ядро
единственный писатель состояния
↔
Agent Gateway
callAgent<T>()
↔
5 агентов
только предложения

Каждый из 5 агентов — узкий одноразовый (single-shot) вызов со строгой JSON-схемой ответа и обязательным детерминированным fallback на стороне ядра. Если агент недоступен, вернул невалидный ответ или "галлюцинирует" — урок не ломается: ядро либо отклоняет предложение агента, либо использует заранее определённый fallback.

2 Agent Gateway (src/core/agents/callAgent.ts)

▾

Все 5 агентов используют одну и ту же общую generic-функцию callAgent<T>() без изменений — это не 5 отдельных интеграций с LLM, а один переиспользуемый шлюз (gateway), через который проходит любой вызов агента.

Собрать запрос (strict tool-use, tool_choice принудительно на единственный tool агента)
↓
Вызвать LLM — модель claude-haiku-4-5, TIMEOUT_MS = 12000 мс на попытку, max_tokens: 512
↓
Отфильтровать ровно 1 блок tool_use
↓
Zod safeParse входных данных tool
↓ при любой ошибке (сеть ИЛИ Zod)
Ровно ОДИН повтор (без задержки/backoff)
↓ при повторной ошибке
Детерминированный fallback агента — source:"core", failed:true

Успешный вызов возвращает source:"agent", failed:false. TIMEOUT_MS был поднят с 8000 до 12000 после живого smoke-тестирования, показавшего реальную задержку Worker'а ~6.4–7 секунд — слишком близко к старому потолку в 8 секунд.

3 Cloudflare Worker proxy (worker/src/index.ts)

▾

Браузер никогда не обращается к LLM-роутеру напрямую. Вместо этого он вызывает развёрнутый Cloudflare Worker по адресу (текстом, не как загружаемый ресурс):

https://english-quest-llm-proxy.leto99999.workers.dev

Worker держит реальный API-ключ только на сервере (секрет платформы env.LLM_API_KEY) — ключ никогда не попадает в бандл браузера — и перенаправляет запрос дальше, на:

https://api.llmrouter.ru

Браузер
→
Cloudflare Worker
хранит ключ, отвечает на OPTIONS
→
api.llmrouter.ru

Worker сам отвечает на CORS OPTIONS preflight запросом 204 без проверки авторизации — сам роутер отклонял preflight запросы с 401 (реальный найденный вживую баг: curl не отправляет preflight, а настоящие браузеры — да).

Две причины существования Worker'а:

  • (a) безопасность — ключ не должен попадать в JS браузера ни при каких условиях;
  • (b) структурное решение CORS-проблемы.

Worker — тонкий reverse-proxy: без системы авторизации, без rate limiting, без прикладной логики.

Дополнительный найденный вживую фикс: заголовок anthropic-dangerous-direct-browser-access (автоматически отправляемый SDK при dangerouslyAllowBrowser: true) пришлось добавить в Access-Control-Allow-Headers Worker'а — он молча блокировал реальный POST-запрос со стороны клиента, хотя OPTIONS возвращал 204 и curl работал.
