# English Quest — спецификация MVP

Консолидированная спецификация для сборки MVP. Самодостаточный документ: содержит цель, архитектуру, логику, данные, агентов, обработку ошибок и визуальный дизайн. Конкретные данные первого урока — в `Lesson-1A.json`.

---

## 1. Обзор и цель

English Quest — интерактивное приложение для детской практики английского языка. Ребёнок проходит короткий урок (теория + задания), приложение гибко проверяет ответы, ведёт прогресс, повторяет слабые темы, начисляет игровые «рубли» и показывает родителю короткий отчёт.

Главная цель MVP — проверить механику обучения: проверка ответов, ошибки, персонализация, повторение, прогресс, бонусы.

Проект делается как дипломная работа; отдельная цель — практика с AI-агентами, поэтому агенты — часть MVP-архитектуры.

## 2. Пользователь и формат

- Основной пользователь: ребёнок уровня Intermediate (материал English File Intermediate, юнит `1A`).
- Интерфейс смешанный: инструкции можно по-русски, упражнения — по-английски.
- Второй пользователь: родитель — видит короткий отчёт после урока.
- Голос, аудио, speaking, микрофон — НЕ входят в MVP.

## 3. Ключевые архитектурные решения

1. **Гибрид: детерминированное ядро + LLM-агенты.** Числа считает код (ядро), суждение и текст даёт агент. Принцип: **агент предлагает — ядро записывает**. Числа никогда не приходят «из головы» LLM.
2. **Входные данные готовы.** Урок собран заранее как `Lesson-1A.json`; шага автоматического разбора PDF и шага «утверждения взрослым» в MVP нет (post-MVP).
3. **Локально в браузере.** Прогресс — в `localStorage`, без бэкенда.
4. **Простые агенты.** 5 независимых агентов как «функции» (один запрос → строгий JSON), без сложной оркестрации. Оркестратор — само ядро, вызывает агента в фиксированных точках.
5. У каждого агента есть детерминированный **fallback** — урок не ломается без LLM.

## 4. Технологический стек

- Frontend: HTML / CSS / JavaScript (детский Roblox-вдохновлённый стиль, без бренда Roblox).
- Хранение: `localStorage`, один ключ `english-quest-progress-v1`.
- Агенты: вызовы LLM (Claude) со структурированным входом и строгим JSON-выходом.
- Без серверной части в MVP.

## 5. Данные урока

Источник — готовый файл `Lesson-1A.json` (схема `lesson-json-v1`). Структура:

- метаданные: `lessonId`, `schemaVersion` (`lesson-json-v1`), `unitTitle` («1A Food: fuel or pleasure?»), `theme`, `lessonGoal`, `status: ready`;
- `targetVocabulary`, `targetGrammar`;
- `theory` — блок теории (см. §11);
- `defaultRewardRules`, `defaultNormalization` — общие правила урока;
- `sections[]` — разделы с упражнениями;
- `reviewExercises[]` — заполняется в рантайме.

Первый урок содержит 3 раздела, 19 упражнений (26 действий ребёнка):
- Grammar (present simple/continuous, action/non-action verbs) — 9 упражнений `text-input`;
- Food vocabulary — 9 упражнений `text-input`;
- Restaurant matching (картинка → слово) — 1 упражнение `matching` на 8 пар.

Типы упражнений MVP: `text-input`, `single-choice`, `matching`, `order-builder`.

Раздел: `sectionId`, `title`, `instructionRu`, `instructionEn`, `skill`, `imagePolicy`, `exercises[]`.

Каждое упражнение: `exerciseId`, `catalogRef`/`catalogItemRef`, `sourceRef` (книга/юнит/страница), `type`, `skill`, `prompt`, `answerCheck` (`correctAnswers` + `acceptedAnswers` или `pairs`), `hint` (`firstError`/`secondError`/`parentExplanation`), `topicImpact`.

## 6. Архитектура: граница ядро/агент

| Что | Исполнитель |
|---|---|
| Хранение (localStorage), баланс рублей, счётчики, пороги статусов | ядро |
| Нормализация текста и точное сравнение ответа | ядро |
| Неоднозначный «похожий» ответ + тип ошибки | агент Answer Checker |
| Интерпретация прогресса, следующий фокус, тексты | агент Progress Advisor |
| Качественные решения по бонусам + похвала | агент Reward Advisor |
| Отчёт родителю | агент Parent Report Generator |
| Объяснение теории (проще, если не понятно) | агент Theory Tutor |

## 7. Детерминированное ядро

Ядро — источник правды, детерминировано и тестируемо:
- хранит данные в `localStorage`;
- нормализует ответ (нижний регистр, trim, схлопывание пробелов, удаление финальной пунктуации) и точно сравнивает с `acceptedAnswers`;
- ведёт счётчики: попытки, правильные, ошибки, серии;
- применяет пороговые правила статусов тем;
- начисляет фиксированные суммы рублей с лимитами;
- перепроверяет любые предложения агента перед записью.

**Модель хранения** (`localStorage`, один ключ `english-quest-progress-v1`):
- `studentProfile` — профиль ученика (см. §12);
- `lessonHistory` — пройденные уроки: результаты упражнений, заработано рублей, снимок слабых тем;
- `topicStats`, `wordStats`, `exerciseTypeStats` — статистика по темам, словам, типам заданий;
- `currentRewards` — текущий баланс рублей;
- `rewardHistory` — события начислений (леджер);
- `reviewQueue` — очередь повторения.

После каждого ответа урок эмитит структурированное событие `exercise_attempt` (id упражнения, номер попытки, `isCorrect`, `errorType`, `topicImpact`, `rewardEvents`), которое обновляет структуры выше и питает родительский отчёт.

## 8. Пять агентов

Каждый агент — один запрос к LLM, строгий JSON на выходе, не трогает числа. Полные скиллы (промпты) с примерами — отдельно; ниже контракты.

1. **Answer Checker** — для `text-input`, когда нет точного совпадения. Вход: prompt, `correctAnswers`, `acceptedAnswers`, ответ ребёнка. Выход: `isCorrect`, `errorType` (typo / wrong_word / wrong_order / missed_article / wrong_tense / non_action_verb_in_continuous / wrong_question_order / missing_auxiliary / spelling_third_person_s / spelling_ing_form / unknown), `confidence`, `hintRu`. Fallback: строгое сравнение, `errorType: unknown`.

2. **Progress Advisor** — вход: `topicStats`, `wordStats`, `exerciseTypeStats`, последние результаты, `difficultyMode`. Выход: `recommendedFocus`, `suggestedDifficulty` (easy/normal/challenge), `reviewSuggestions`, `motivationalMessageRu`, `sessionAdvice` (continue/soft_finish). Fallback: только пороговые правила ядра.

3. **Reward Advisor** — вход: результат проверки, номер попытки, история, серия. Выход: `honestAttempt`, `looksLikeSpam`, `suggestedReasons` (из разрешённого списка), `celebrationRu`. Суммы задаёт ядро. Fallback: ядро само применяет reward-правила.

4. **Parent Report Generator** — вход: снимок урока (пройдено, верно, ошибки, слабые темы, рубли) + рекомендация. Выход: `parentReportRu`, `headlineRu`. Fallback: шаблонный отчёт.

5. **Theory Tutor** — показывает теорию; на «не понятно» объясняет проще. Вход: правило, текущий уровень, `childUnderstood`, число кругов. Выход: `explanationRu`, `exampleRu`, `level` (normal/simple/simplest), `canSimplifyMore`. Fallback: заранее написанные уровни из `theory.explanationLevels`.

## 9. Логика проверки ответов и прогресса

`checkAnswer()`:
1. ядро нормализует ответ;
2. `matching`/`single-choice`/`order-builder` — детерминированное сравнение по id / собранной строке;
3. `text-input` — точное сравнение с `acceptedAnswers`; при совпадении вердикт без агента;
4. иначе — вызов Answer Checker (вердикт + `errorType`);
5. если агент недоступен — строгий fallback.

После каждого ответа ядро обновляет счётчики, `topicStats`, `wordStats`, `exerciseTypeStats`.

Слабые места (пороги — ядро):
- 1 ошибка по теме — показать подсказку;
- 2 ошибки — статус темы `Повторить` + добавить задания в `reviewQueue`;
- 3+ ошибки — расширить мини-тренировку;
- 3 правильных подряд — усилить статус;
- успешное повторение — приблизить к `Выучено`.

Машина статусов темы: `Не изучено → В процессе → Повторить → Выучено` (тема может вернуться в `Повторить`).

Очередь повторения: та же тема/слова/грамматика, близкий тип, готовое задание из урока, ещё не пройденное в сессии. Генерация новых заданий — post-MVP.

## 10. Игровые рубли (бонусы)

Награда за учебное действие, а не только за идеальный ответ. Ошибка не наказывается. Суммы фиксированы в ядре:

| Причина | Сумма | Лимит |
|---|---|---|
| honest_attempt | +1 ₽ | 1 раз/задание |
| first_try_correct | +5 ₽ | 1 раз/задание |
| correct_after_hint | +3 ₽ | 1 раз/задание |
| fixed_mistake | +4 ₽ | 1 раз/задание |
| streak_bonus | +10 ₽ | за серию из 5 |
| weak_topic_closed | +15 ₽ | при закрытии слабой темы |

Правила: `first_try_correct` и `correct_after_hint` взаимоисключающие; `fixed_mistake` — только после предыдущего неверного; нет фарма на одном задании. Reward Advisor предлагает причины и пишет похвалу — ядро проверяет и начисляет. Каждое начисление — событие в леджере (`rewardEventId`, `reason`, `amount`, `attemptNumber`, `createdAt`, `relatedTopic`/`relatedWord`).

## 11. Блок теории (шаг перед заданиями)

В начале урока — короткая теория с кнопками «понятно / не понятно». Блок `theory` в уроке содержит правило и заранее написанные уровни (`normal`, `simple`). На «не понятно» ядро показывает следующий уровень, затем зовёт Theory Tutor для более простого объяснения. Число кругов ограничено `maxSimplifyRounds` (3), потом мягкий переход к практике. Кнопки и счётчик кругов — ядро.

## 12. Профиль ученика и персонализация

`studentProfile` (в `localStorage`): `studentId` (в MVP `primary`), `displayName`, `levelLabel`, `preferredSessionLength`, `confidenceScore`, `motivationSignals`, `difficultyMode` (easy/normal/challenge), `lastRecommendedFocus`.

- `confidenceScore` — простая формула ядра: `clamp(correctRatio + 0.05*streak - 0.1*errorsInARow, 0, 1)`.
- `difficultyMode` меняет ядро по совету Progress Advisor, но через защитные правила (не прыгать easy→challenge напрямую; менять только между уроками; вверх — после 3 правильных подряд, вниз — после 2 ошибок или усталости).
- Сигналы мотивации: серия, использованные подсказки, исправления после подсказки, ошибки подряд, длинная сессия. При усталости/многих ошибках — предложить мягкое завершение.

## 13. Родительский отчёт

После урока Parent Report Generator собирает короткий текст: сколько пройдено, сколько верно, что даётся трудно, что повторить, сколько рублей, одна рекомендация. Тон: просто, честно, с поддержкой, без терминов. Fallback — шаблон из тех же полей.

## 14. Обработка ошибок и fallback

- Ответ агента не доверенный, пока ядро не проверит: валидный JSON ожидаемой формы, значения из допустимых наборов.
- При сбое (битый JSON / таймаут / недоступность): один повтор → детерминированный fallback агента.
- В событиях фиксируется `source` (core/agent) и факт сбоя — для отладки.

## 15. Визуальный стиль и экран урока

Стиль: детский, блочный, яркий, крупные скруглённые кнопки — Roblox-вдохновлённый, без бренда, логотипов и ассетов Roblox.

Экран урока сверху вниз:
- верхняя панель: название, баланс рублей, прогресс («задание 5 из 19»);
- заголовок урока: юнит, тема, цель;
- (шаг 0) экран теории: правило + пример + кнопки «понятно / не понятно»;
- карточка задания: метка раздела, инструкция RU + EN, задание, поле ввода/выбор, крупная кнопка «Проверить».

Состояния: верно (зелёное + рубли), ошибка (спокойная подсказка + «попробуй ещё»), matching (картинки + слова), повторение слабой темы. Экран родителя: одна строка-итог + короткий текст.

## 16. Что НЕ входит в MVP (post-MVP)

Не входят: голос/произношение, listening, reading как режим, автоматический разбор PDF, полный импорт учебника и Test Book, шаг утверждения взрослым, магазин/аватары, несколько учеников, генерация новых заданий, глубокая персонализация, прочие агенты (Exercise Generator и т.п.) и мультиагентная оркестрация.

## 17. Пайплайн одного урока

1. ядро берёт готовый `Lesson-1A.json`;
2. ядро показывает теорию (при «не понятно» — Theory Tutor);
3. ядро рендерит задания;
4. проверка ответа: ядро (нормализация/точное сравнение) + Answer Checker при неоднозначности;
5. ядро пишет прогресс и статусы;
6. Progress Advisor даёт следующий фокус и сложность;
7. Reward Advisor предлагает бонусы → ядро начисляет суммы;
8. ядро наполняет очередь повторения;
9. Parent Report Generator собирает отчёт родителю.

## 18. Критерии успеха (приёмка)

MVP собран, если можно вручную проверить:
1. загрузить урок из `Lesson-1A.json`, пройти теорию;
2. пройти минимум 10 заданий, сделать 2+ ошибки в одной теме;
3. увидеть сохранение прогресса после перезагрузки страницы;
4. получить упражнение в `reviewQueue` и пройти повторение;
5. увидеть изменение `studentProfile` и `topicStats`;
6. увидеть начисления в `rewardHistory` и рост баланса;
7. открыть короткий родительский отчёт без обращения к серверу;
8. при недоступности агента — работающий fallback.
