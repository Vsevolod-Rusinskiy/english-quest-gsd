1 Обзор архитектуры (core/agent split)

▾

Центральный тезис проекта (сформулирован в CLAUDE.md): агент предлагает — ядро проверяет перед использованием. Детерминированное ядро (core) — единственный писатель состояния и единственный владелец всех чисел в приложении. Ровно 5 независимых LLM-агентов-функций — Answer Checker, Theory Tutor, Reward Advisor, Progress Advisor, Parent Report Generator — только предлагают текст или суждение (подсказку, похвалу, рекомендацию), но никогда не пишут состояние напрямую.

Детерминированное ядро
единственный писатель состояния
↔
Agent Gateway
callAgent<T>()
↔
5 агентов
только предложения

Каждый из 5 агентов — узкий одноразовый (single-shot) вызов со строгой JSON-схемой ответа и обязательным детерминированным fallback на стороне ядра. Если агент недоступен, вернул невалидный ответ или "галлюцинирует" — урок не ломается: ядро либо отклоняет предложение агента, либо использует заранее определённый fallback.

2 Agent Gateway (src/core/agents/callAgent.ts)

▾

Все 5 агентов используют одну и ту же общую generic-функцию callAgent<T>() без изменений — это не 5 отдельных интеграций с LLM, а один переиспользуемый шлюз (gateway), через который проходит любой вызов агента.

Собрать запрос (strict tool-use, tool_choice принудительно на единственный tool агента)
↓
Вызвать LLM — модель claude-haiku-4-5, TIMEOUT_MS = 12000 мс на попытку, max_tokens: 512
↓
Отфильтровать ровно 1 блок tool_use
↓
Zod safeParse входных данных tool
↓ при любой ошибке (сеть ИЛИ Zod)
Ровно ОДИН повтор (без задержки/backoff)
↓ при повторной ошибке
Детерминированный fallback агента — source:"core", failed:true

Успешный вызов возвращает source:"agent", failed:false. TIMEOUT_MS был поднят с 8000 до 12000 после живого smoke-тестирования, показавшего реальную задержку Worker'а ~6.4–7 секунд — слишком близко к старому потолку в 8 секунд.

3 Cloudflare Worker proxy (worker/src/index.ts)

▾

Браузер никогда не обращается к LLM-роутеру напрямую. Вместо этого он вызывает развёрнутый Cloudflare Worker по адресу (текстом, не как загружаемый ресурс):

https://english-quest-llm-proxy.leto99999.workers.dev

Worker держит реальный API-ключ только на сервере (секрет платформы env.LLM_API_KEY) — ключ никогда не попадает в бандл браузера — и перенаправляет запрос дальше, на:

https://api.llmrouter.ru

Браузер
→
Cloudflare Worker
хранит ключ, отвечает на OPTIONS
→
api.llmrouter.ru

Worker сам отвечает на CORS OPTIONS preflight запросом 204 без проверки авторизации — сам роутер отклонял preflight запросы с 401 (реальный найденный вживую баг: curl не отправляет preflight, а настоящие браузеры — да).

Две причины существования Worker'а:

  • (a) безопасность — ключ не должен попадать в JS браузера ни при каких условиях;
  • (b) структурное решение CORS-проблемы.

Worker — тонкий reverse-proxy: без системы авторизации, без rate limiting, без прикладной логики.

Дополнительный найденный вживую фикс: заголовок anthropic-dangerous-direct-browser-access (автоматически отправляемый SDK при dangerouslyAllowBrowser: true) пришлось добавить в Access-Control-Allow-Headers Worker'а — он молча блокировал реальный POST-запрос со стороны клиента, хотя OPTIONS возвращал 204 и curl работал.

4 Пять агентов

▾

Каждый агент — узкая одноразовая функция поверх общего Agent Gateway. Ниже — что каждый предлагает, что его триггерит, каков детерминированный fallback и какая проверка/guardrail стоит на стороне ядра.

Answer Checker

Триггер
Только когда детерминированная точная проверка (exact-match) text-input упражнения провалилась. Никогда не вызывается для single-choice / matching / order-builder — там проверка всегда полностью детерминированная, без агента.
Предлагает
isCorrect, errorType, confidence, hintRu
Core-проверка
Trust gate: CORRECT_CONFIDENCE_THRESHOLD = 0.8 — ядро принимает isCorrect:true от агента ТОЛЬКО при confidence >= 0.8; ниже порога ответ понижается до isCorrect:false (подсказка всё равно передаётся).
Fallback
isCorrect:false, общий errorType:"unknown", без confidence/hint (агент "не говорил").

Theory Tutor

Триггер
Нажатие ребёнком "Не понятно" на экране теории.
Предлагает
Упрощённую формулировку объяснения теории.
Core-проверка
maxSimplifyRounds = 3 (из данных урока). Раунд 1 (simplifyRoundCount === 0) — ТОЛЬКО ядро, без вызова агента: отдаётся заранее написанный текст explanationLevels[1] ("простой"). Раунды 2–3 — вызов агента для ещё более простой переформулировки. Мягкий переход: когда счётчик раундов достигает maxSimplifyRounds (3), theoryUnderstood становится true независимо от последнего нажатия "Не понятно" — автоматический переход к первому упражнению (предотвращает бесконечный цикл). У этого перехода пока нет поясняющей UI-подсказки — известный, ещё не закрытый UX-нюанс.
Fallback
Повторно отдаёт ТОТ ЖЕ заранее написанный простой текст дословно — никогда не выдумывает новый.

Reward Advisor

Триггер
Один раз на каждый ответ, ТОЛЬКО когда этот ответ уже принёс ≥1 награду (правила награды ядра уже решили, что рубли начислены).
Предлагает
celebrationRu (похвала) и suggestedReasons. Агент НИКОГДА не решает сумму — у него нет поля amount.
Core-проверка
Анти-галлюцинационный gate: suggestedReasons агента пересекаются (пересечение множеств) с РЕАЛЬНО начисленными delta.rewardEvents. Если агент предложил причину, которой не было — praiseRu становится undefined, идентично сбою агента. praiseRu — временное значение (возвращается из handleAnswer, никогда не сохраняется).
Fallback
Без текста похвалы.

Progress Advisor

Триггер
Один вызов в конце сессии.
Предлагает
recommendedFocus, suggestedDifficulty, reviewSuggestions, motivationalMessageRu, sessionAdvice.
Core-проверка
ДВА guardrail'а на стороне ядра (оба — реальные найденные вживую баги, случай "доверяй, но проверяй"): (a) applyDifficultyGuardrails() — единственный писатель difficultyMode; ограничивает прыжок ровно ОДНИМ шагом за вызов (без прямого easy↔challenge); повышение требует correctStreak >= 3, понижение — recentErrors >= 2, иначе без изменений. (b) applyRecommendedFocusGuardrail() — проверяет recommendedFocus по известному набору id тем из topicLabels.ts; при несовпадении откатывается на собственную вычисленную ядром самую слабую тему. Это закрыло реальный баг, когда агент вернул галлюцинированную строку на смеси языков вместо чистого id темы.
Fallback
Использует собственную самую слабую тему ядра (наивысшее отношение ошибок/наименьшее отношение верных в topicStats), сложность не меняется, фиксированное сообщение поддержки.

Parent Report Generator

Триггер
Вызывается СРАЗУ ПОСЛЕ разрешения Progress Advisor — последовательно, явно НЕ Promise.all — чтобы безопасно использовать уже прошедший guardrail финальный recommendedFocus Progress Advisor'а.
Предлагает
parentReportRu, headlineRu.
Fallback
buildTemplateReport() — детерминированный русский шаблон из снимка сессии (пройденные упражнения, число верных ответов, темы для повторения/проблемные через topicLabel(), заработанные рубли, рекомендация). Этот шаблон сам был местом реального бага (когда-то подставлял сырые id тем ещё до появления topicLabels.ts) — сейчас исправлено.

5 Поток проверки ответа

▾

Самый частый поток в приложении — lessonEngine.handleAnswer.

Детерминированная проверка по типу: checkTextInput / checkSingleChoice / checkMatching / checkOrderBuilder
↓
Только text-input эскалирует к Answer Checker при промахе точного совпадения; остальные типы — полностью детерминированы, без агента
↓
Единый чистый проход evaluateAttempt(): считает всё производное состояние (статы по темам + FSM, очередь повторения, reward-события, статы слов/типов упражнений, серии)
↓
Reward Advisor вызывается условно (если ≥1 reward-событие)
↓
Один dispatch exercise_attempt
↓
Сохранение в localStorage["english-quest-progress-v1"]
↓
UI перерисовывается (баннер обратной связи, чипы top-bar, прогресс-бар)

6 Освоение тем / очередь повторения (FSM)

▾

Логика в topicStatusMachine.ts + reviewQueue.ts — чистая логика ядра, ноль участия агентов.

not_started
→
in_progress
→
needs_review
2 ошибки суммарно
любое состояние
→
mastered
3 верных ответа подряд

Упражнения темы в статусе needs_review, на которые не было недавнего верного ответа, добавляются в очередь повторения и повторно показываются позже В ТОЙ ЖЕ сессии урока (проход повторения после основного прохода). Ответ на проходе повторения убирает элемент из очереди независимо от правильности.

Строгое правило (важная оговорка): если тема хоть раз накопила ≥2 ошибок за всю историю, один последующий неверный ответ мгновенно возвращает её в needs_review даже после того, как она была освоена (mastered).

7 Механика наград

▾

Фиксированная таблица сумм наград определена в ядре (rewardEngine.ts) — сумму никогда не выбирает агент:

СобытиеСумма, ₽
honest_attempt1
correct_after_hint3
fixed_mistake4
first_try_correct5
streak_bonus10
weak_topic_closed15

streak_bonus (+10) срабатывает, когда сессионная серия верных ответов подряд достигает >= 5 (затем сбрасывается).

Синтезированный звук "звона монет" (src/ui/sound/coin.ts, Web Audio API, без внешнего аудиофайла — две колокольчик-подобные ноты: B5 ~988 Гц, затем E6 ~1319 Гц) проигрывается при увеличении баланса рублей; деградирует бесшумно (никогда не выбрасывает исключение), если AudioContext недоступен/заблокирован.

Чип серии "🔥 N" отображается только когда currentCorrectStreak >= 2.

9 Почему такая архитектура

▾

Цель диплома — доказать механику обучения целиком: детерминированная проверка ответов + LLM-интерпретация там, где она действительно нужна + персонализация по прогрессу + повторение слабых тем + начисление рублей — без единого «сломанного» состояния, даже когда агент недоступен.

У каждого из 5 агентов есть рабочий детерминированный fallback, проверенный вживую: таймауты, сбои CORS и галлюцинации наблюдались на практике и обрабатывались без падения урока.

Поймано вживую

  • CORS preflight 401 → исправлено: Worker сам отвечает на OPTIONS (+ фикс заголовка anthropic-dangerous-direct-browser-access).
  • Сырой snake_case id темы утекал в текст для ребёнка/родителя → исправлено через topicLabels.ts.
  • Отсутствовал core-side guardrail на recommendedFocus (агент вернул галлюцинированную строку на смеси языков) → исправлено через applyRecommendedFocusGuardrail().
  • Таймаут 8с был слишком тесным против реальной задержки ~6.4с+ → поднят до 12с (TIMEOUT_MS).
English Quest — диплом-проект. Документ сгенерирован для объяснения актуальной архитектуры (детерминированное ядро + 5 LLM-агентов). Открывается напрямую через file://, без сборки, без сети.
